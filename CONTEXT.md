# CareBook 域模型

本文档是 AI Agent 的业务语言入口，定义核心业务对象及其关系。详细规则、流程、实施方案见对应子文档，避免重复。

---

## 核心业务对象

### 用户（User）

C 端账号主体。职责：管理被护理人档案、发起订单、查看护理日报。一个用户可绑定多个被护理人。

### 被护理人（CareReceiver）

护理服务的实际承接对象，护理日报的内容主体。每个被护理人当前归属于一个用户（`owner_user_id`）。线下反向激活允许覆盖归属用户，但必须记录变更日志。

### 护工（Caregiver）

护理服务的实际提供者。护理员身份复用系统既有的 `caregiver` RBAC 角色，不创建独立登录账号；统一小程序按当前登录用户的角色决定菜单和工作台入口。护工资料（从业年限、擅长、性格标签等）不塞进用户主表。

**权限规则：**
- `caregiver` 角色决定护理员菜单和页面入口是否可见；角色不匹配时不展示护理员工作台。
- 后端接口先校验 `caregiver` 角色，再校验具体服务关系的 `caregiver_user_id` 是否为当前用户，角色可见不等于可以操作其他护理员的任务。

### 智能工牌（Badge Device）

护工用于采集服务证据的工作设备，不是可被派单的履约主体。设备必须绑定到护工；订单、服务关系和家属日报均以护工为业务归属，再通过护工的设备绑定历史路由录音、转写和 AI 证据。

### 服务关系（CareRelation）

护理业务的履约主轴。明确"哪位护工服务哪位被护理人"，是护理日报的底层履约来源。

**强规则：**
- 同一订单同一时刻只能有 1 条有效履约关系
- 同一被护理人同一时刻只能有 1 条 `serving` 状态的服务关系
- 改派必须走"关旧开新"原子过程
- 每条服务关系必须指派 1 名护工，`care_relation.caregiver_user_id` 必填
- 派单护工必须同时具备 `caregiver` 角色、在用的护理员扩展档案和有效工牌绑定
- 设备不参与派单，不得作为 `care_relation` 或订单上的可选履约主体
- 日报证据按服务时间段关联护工的设备绑定历史；护工无有效设备证据是“待审核/不推送”的业务状态，不得降级为设备直接履约

### 服务（Service）

可购买的服务对象。服务时长类型：`hour`（按小时）、`day`（按自然日）、`month`（按自然月）。

### 订单（Order）

交易来源模块，不是最终服务事实模块。分线上订单（`care_online_order`）和线下订单（`care_offline_order`）。

### 服务时间（Service Timing）

同一订单的时间语义必须分开：用户预约时间是下单时提出的原始诉求和退款依据，不因运营排班而覆盖；运营与家属沟通后，在服务关系上填写计划上门与结束时间，作为内部排班和通知护工的依据；实际上单、下单时间才是履约事实和结算依据。

### 护理日报（NurseDailyReport）

对用户可见的正式业务结果。按"同一订单 + 同一自然日"聚合生成，次日早上统一产出。

## 对象关系图

```
User（用户）
  └── owner_user_id ──→ CareReceiver（被护理人）
                            │
                            └── CareRelation（服务关系）←── caregiver_user_id
                                     │
                                     └── Order（订单：线上或线下）
                                              │
                                              └── NurseDailyReport（护理日报）
```

---

## 关键约定

1. 软删除：`delFlag = '0'` 正常，`'1'` 已删
2. 金额：整数分存储（如 `30000` = 300 元）
3. 响应：`{ code, msg, data }`，HTTP 状态码始终 200
4. 时间：使用 dayjs，禁止拼接字符串
5. 统一小程序菜单按 RBAC `roleKey` 过滤；前端菜单隐藏不代替后端权限校验
