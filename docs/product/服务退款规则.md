# 服务退款规则

本文档定义 CareBook 线上护理订单取消后的退款口径。当前文档既作为产品规则，也反映当前阶段已经落地的退款实现边界。

## 1. 当前实现现状

当前线上订单取消与退款链路已经具备基础闭环：

1. 用户或后台调用取消后，`care_online_order.order_status` 更新为 `cancelled`。
2. 如果订单未支付，`payment_status` 更新为 `closed`。
3. 如果订单已支付，会创建退款申请单 `payment_refund_order`，并计算退款建议。
4. 命中标准低风险规则时，系统会自动发起微信退款。
5. 微信退款结果通过回调主驱动更新，并由查询补偿任务兜底收敛状态。

当前已经不是“只有取消没有退款”的半闭环；后续仍需继续完善复杂责任归属、优惠拆分和更细粒度自动化规则。

## 2. 适用范围

本规则优先适用于小程序 C 端线上订单，即 `care_online_order`。

线下订单 `care_offline_order` 通常走机构、保司或线下结算，不默认套用本规则。线下订单如果后续接入线上收款，应单独定义渠道结算和退款规则。

## 3. 核心定义

### 3.1 关键时间

| 名称 | 字段或来源 | 说明 |
| --- | --- | --- |
| 取消申请时间 | 系统当前时间 | 用户或运营发起取消的时间 |
| 预约开始时间 | `planned_start_at` | 用户预约服务开始时间 |
| 实际开始时间 | `care_relation.actual_start_at` | 护工实际上单或服务实际开始时间 |
| 抵达现场时间 | 待扩展 | 护工到达但服务未开始的场景，当前可由运营人工确认 |

时间判断优先级：

1. 如果已有 `actual_start_at`，说明服务已开始，按服务中或服务后规则处理。
2. 如果没有 `actual_start_at`，使用 `planned_start_at` 判断距服务开始还有多久。
3. 如果缺少 `planned_start_at`，不能自动退款，进入人工审核。

### 3.2 责任归属

| 归属 | 说明 |
| --- | --- |
| 用户原因 | 用户个人原因取消、临时无法配合、地址或联系人问题、现场爽约等 |
| 平台原因 | 无人接单、超时未派单、平台调度失败、服务描述与实际不符等 |
| 服务人员原因 | 护工无法履约、迟到严重、服务人员原因导致服务无法进行等 |
| 误操作 | 用户误下单，且未造成调度或服务损失 |
| 特殊优惠 | 使用代金券、兑换券、特殊权益等非现金或强限制订单 |

责任归属必须可追溯。自动判断不足时，由运营在后台选择原因并记录备注。

## 4. 退款类型

退款结果分三类：

| 类型 | 退款金额 | 订单表现 |
| --- | --- | --- |
| 全额退款 | 退还已支付金额 100% | 订单取消，退款完成后 `refund_status = refunded` |
| 部分退款 | 按阶梯退还已支付金额，扣除 3% / 5% / 8% 平台服务费 | 订单取消或终止，退款完成后 `refund_status = part_refunded` |
| 不退款 | 退款金额为 0 | 订单取消或完成，`refund_status = none` 或后续扩展为 `refund_rejected` |

金额单位统一为分。计算部分退款时：

```text
service_fee_amount = round(paid_amount * 8%)
refund_amount = paid_amount - service_fee_amount
```

如后续存在优惠券、余额、组合支付或分账，必须先定义各支付资金来源的退款优先级。

## 5. 全额退款规则

满足任一条件时，可全额退款。

### 5.1 平台或服务人员原因

以下场景全额退款：

1. 无人接单。
2. 平台超时未派单，导致无法按约服务。
3. 平台主动取消。
4. 服务描述与实际服务能力明显不符。
5. 护工或服务人员原因导致无法履约。
6. 服务人员严重迟到且用户拒绝继续服务。

平台或服务人员原因不扣平台服务费。如果用户已产生额外损失，是否补偿不在本规则内，需另行定义售后补偿规则。

### 5.2 用户提前取消

用户因个人原因取消，但取消时间距离预约开始时间超过 24 小时：

```text
planned_start_at - cancel_requested_at > 24 小时
```

处理结果：

1. 全额退款。
2. 记录一次用户主动取消，但不计入扣费取消次数。

### 5.3 误操作

用户误下单且满足以下条件时，可全额退款：

1. 未派单，或虽已派单但未产生明确调度成本。
2. 未开始服务。
3. 运营确认属于误操作。

误操作是否计入当月取消次数，由运营规则决定。默认不计入扣费取消次数。

## 6. 部分退款规则

满足条件时，退款金额按阶梯扣除平台服务费。

### 6.1 用户原因取消的阶梯扣费

用户因个人原因取消且服务尚未开始时，按距离预约开始时间做阶梯扣费：

| 距离预约开始时间 | 退款规则 | 平台服务费 |
| --- | --- | --- |
| `> 24 小时` | 全额退款 | 0% |
| `12-24 小时` | 部分退款 | 3% |
| `2-12 小时` | 部分退款 | 5% |
| `0-2 小时` | 部分退款 | 8% |

计算口径：

```text
hours_until_service = planned_start_at - cancel_requested_at
service_fee_amount = round(paid_amount * service_fee_rate)
refund_amount = paid_amount - service_fee_amount
```

说明：

1. `hours_until_service <= 0` 时，不进入阶梯退款，按服务已开始或现场爽约规则处理。
2. 阶梯扣费只适用于用户原因取消。
3. 平台原因、服务人员原因、误操作不适用阶梯扣费。
4. 阶梯扣费场景计入当月用户主动取消次数。

### 6.2 用户临近服务取消

用户因个人原因在服务开始前 2 小时内取消：

```text
0 < planned_start_at - cancel_requested_at <= 2 小时
```

处理结果：

1. 扣除已支付金额的 8% 作为平台服务费。
2. 退还剩余 92%。
3. 计入当月用户主动取消次数。

### 6.3 当月频繁取消

用户当月预约成功后，因个人原因取消累计超过 3 次，后续取消开始扣费。

建议口径：

1. 统计周期：自然月。
2. 统计对象：当前登录用户 `user_id`。
3. 只统计预约成功后的用户原因取消。
4. 平台原因、服务人员原因、误操作不计入。
5. 第 1 至第 3 次用户原因取消不因频繁取消扣费。
6. 第 4 次及之后的用户原因取消，按部分退款处理。

待确认点：

1. “预约成功”是指支付成功 `paid`，还是派单成功 `pending_service`。
2. 频繁取消是否按用户统计，还是按被护理人统计。
3. 频繁取消扣费是否覆盖“24 小时外取消”的全额退款规则。

默认建议：预约成功按支付成功计算；频繁取消按用户统计；第一阶段命中频繁取消时先进入人工审核，不直接覆盖阶梯退款结果。

## 7. 不退款规则

满足以下条件时，不退款。

### 7.1 现场爽约或用户阻碍履约

服务人员已经抵达服务现场或服务已经开始，但因用户原因导致无法履约：

1. 联系不上用户或联系人。
2. 用户拒绝开门、拒绝配合服务。
3. 用户临时变更地址导致服务无法按约进行。
4. 用户提供错误地址或错误联系方式，且服务人员已到达或已产生明确履约成本。
5. 服务开始后，用户单方面终止服务且无平台或服务人员责任。

处理结果：

1. 不退款。
2. 订单可标记为 `cancelled` 或按实际服务情况标记为 `completed`，具体由运营确认。
3. 必须记录现场证据或运营备注。

### 7.2 特殊优惠订单

使用代金券、兑换券、特殊权益或明确标记不可退的优惠订单，默认不退款。

待确认点：

1. 代金券是否允许原路退回。
2. 现金补差部分是否可退。
3. 优惠订单是否允许运营特批退款。

默认建议：现金实付部分仍可按规则评估，非现金权益不退现金。

## 8. 状态流转建议

### 8.1 订单状态

取消后订单状态进入：

```text
order_status = cancelled
```

服务已经开始且只退剩余未服务部分时，订单可保持：

```text
order_status = completed
refund_status = part_refunded
```

### 8.2 支付状态

支付状态只表达收款状态，不应直接表达退款细节：

1. 未支付取消：`payment_status = closed`
2. 已支付取消且待退款：`payment_status = paid`，`refund_status = refunding`
3. 全额退款完成：`payment_status = paid`，`refund_status = refunded`
4. 部分退款完成：`payment_status = paid`，`refund_status = part_refunded`

默认建议：支付状态保持支付主状态，退款由 `refund_status` 和退款单表达，避免 `payment_status` 同时承载支付与退款两个维度。

### 8.3 退款状态

当前已有状态：

1. `none`：无退款
2. `part_refunded`：部分退款
3. `refunded`：已退款

建议扩展：

1. `pending_review`：待退款审核
2. `refunding`：退款中
3. `refund_failed`：退款失败
4. `refund_rejected`：审核拒绝退款

## 9. 后端实现建议

后续代码实现不应在 `cancel()` 方法里直接调用微信退款。建议拆成以下步骤：

1. `cancelOrder`：只负责取消申请、权限校验和订单状态初步变更。
2. `evaluateRefundPolicy`：根据时间、责任归属、优惠类型、取消次数计算退款类型和金额。
3. `createRefundRequest`：创建退款申请或退款单，记录规则命中原因。
4. `approveRefund`：运营审核或系统自动审核通过。
5. `executeWechatRefund`：调用微信退款接口。
6. `handleWechatRefundNotify`：处理微信退款回调，幂等更新退款单和订单状态。

建议新增退款单表，避免把所有退款过程字段塞进 `care_online_order`：

```text
payment_refund_order
```

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `payment_order_id` | 支付单 ID |
| `business_type` | 业务类型，如 `care_order` |
| `business_order_id` | 业务订单 ID |
| `refund_no` | 系统退款单号 |
| `wechat_refund_id` | 微信退款单号 |
| `refund_type` | `full/partial/none` |
| `refund_reason_type` | 取消原因分类 |
| `responsibility_party` | 责任归属 |
| `paid_amount` | 原支付金额，分 |
| `refund_amount` | 退款金额，分 |
| `service_fee_amount` | 扣除服务费，分 |
| `refund_status` | 退款单状态 |
| `rule_snapshot` | 命中的退款规则快照 |
| `reviewer_user_id` | 审核人 |
| `reviewed_at` | 审核时间 |
| `raw_notify_body` | 微信退款回调原文 |

## 10. 小程序交互建议

用户点击取消订单时，不应只弹一个确认框。建议根据订单状态展示不同文案：

1. 未支付订单：提示“取消后订单将关闭，无需退款”。
2. 已支付且服务开始前 24 小时以上：提示“预计可全额退款，实际结果以平台审核为准”。
3. 已支付且服务开始前 12-24 小时：提示“可能扣除 3% 平台服务费”。
4. 已支付且服务开始前 2-12 小时：提示“可能扣除 5% 平台服务费”。
5. 已支付且服务开始前 2 小时内：提示“可能扣除 8% 平台服务费”。
6. 服务中或服务人员已抵达：提示“需平台核实后处理，可能无法退款”。
7. 特殊优惠订单：提示“该订单涉及优惠权益，退款规则以活动说明为准”。

取消后订单详情页必须展示：

1. 取消原因。
2. 退款状态。
3. 预计退款金额或“待审核”。
4. 退款失败时的处理提示。

## 11. 待解决问题

以下问题需要产品或运营确认后再进入代码实现：

1. 频繁取消的“预约成功”按支付成功还是派单成功计算。
2. 当月取消次数按用户、被护理人还是手机号统计。
3. “24 小时外取消”和“当月超过 3 次取消”冲突时，哪个规则优先。
4. 误操作允许的时间窗口，例如下单后 5 分钟内。
5. 服务人员“抵达现场”的系统依据，是运营手工确认、定位、打卡还是服务关系状态。
6. 特殊优惠订单中现金实付部分是否允许退款。
7. 微信退款失败后的人工处理流程。
8. 是否允许运营特批覆盖规则，以及特批权限和审计要求。
9. 部分服务已完成后的按比例退款公式。

## 12. 默认规则优先级

在没有额外运营特批时，建议按以下优先级判断：

1. 平台或服务人员原因：全额退款。
2. 特殊优惠不可退规则：不退款或仅退现金实付部分。
3. 服务已开始或现场爽约：不退款。
4. 用户原因且服务开始前 2 小时内：部分退款，扣 8%。
5. 用户原因且服务开始前 2-12 小时：部分退款，扣 5%。
6. 用户原因且服务开始前 12-24 小时：部分退款，扣 3%。
7. 用户原因且服务开始前 24 小时以上：全额退款。
8. 用户原因且当月取消次数超过 3 次：进入人工审核，运营确认是否覆盖阶梯退款。
9. 规则无法判断：进入人工审核。

## 13. 第一阶段实施切片

当前阶段采用“保守自动化 + 人工审核混合模式”。

当前核心原则是：低风险、规则明确的场景允许系统自动执行退款；存在运营或资金风险的不确定规则统一进入 `pending_review`，由运营审核处理。

### 13.1 第一阶段先做

1. 新增退款单模型 `payment_refund_order`。
2. 已支付线上订单取消时，不直接改成已退款，而是生成退款申请。
3. 根据当前可自动判断的数据，给出退款建议并按规则分流：
   - 未支付：直接关闭订单，无退款单。
   - 已支付且缺少预约时间：进入待审核。
   - 已支付且服务已开始：进入待审核，默认不自动退款。
   - 已支付且距离预约开始 24 小时以上：自动全额退款。
   - 已支付且距离预约开始 12-24 小时：建议部分退款，扣 3%。
   - 已支付且距离预约开始 2-12 小时：建议部分退款，扣 5%。
   - 已支付且距离预约开始 2 小时内：建议部分退款，扣 8%。
4. 后台能看到退款申请、建议退款金额、命中规则、待审核原因。
5. 小程序订单详情能看到退款状态：
   - 待审核
   - 退款中
   - 已全额退款
   - 已部分退款
   - 退款失败
   - 不予退款

### 13.2 第一阶段暂不做

1. 不处理优惠券、代金券、权益券的资金拆分。
2. 不做按已服务时长的比例退款。
3. 不做运营特批权限细分。
4. 不做退款失败后的财务人工打款流程。

### 13.3 需要产品确认后再写死的规则

以下规则不能直接写死进代码，先用“待审核”兜底：

1. 当月超过 3 次取消是否覆盖 24 小时外全额退款。
2. 误操作的时间窗口。
3. 特殊优惠订单的现金部分是否可退。
4. 服务人员已抵达但服务未开始的系统判定依据。

### 13.4 待定产品口径的第一阶段处理

| 待定产品口径 | 第一阶段临时逻辑 | 后续可能规则 |
| --- | --- | --- |
| 服务开始前 12-24 小时用户原因取消 | 自动建议部分退款，扣 3% | 后续可根据运营成本调整为人工审核或其他比例 |
| 服务开始前 2-12 小时用户原因取消 | 自动建议部分退款，扣 5% | 后续可根据运营成本调整为人工审核或其他比例 |
| 当月频繁取消是否覆盖 24 小时外全额退款 | 即使命中 24 小时外全额条件，只要当月取消超过 3 次，先进入 `pending_review` | 频繁取消优先级可能更高；第 4 次及之后用户原因取消大概率扣 8%，防止恶意刷单或反复占用服务资源 |
| 误下单窗口多久 | 用户申诉误下单时进入 `pending_review`，由运营判断是否全额退款 | 后续可设硬窗口，例如下单后 5 分钟内且未接单，允许用户自主全额取消 |
| 特殊优惠订单现金部分是否可退 | 含优惠券、代金券、权益券的订单统一进入 `pending_review` | 现金实付部分按规则退回；非现金权益可原路退回且不顺延有效期，或按活动规则作废 |

说明：阶梯退款比例作为第一阶段可实现规则；频繁取消、误下单、特殊优惠仍作为产品预判，不作为第一阶段自动最终判定依据。第一阶段实现时，凡是命中审核场景，都应记录规则命中原因并进入待审核。

### 13.5 推荐状态机

退款单状态建议：

```text
pending_review -> approved -> refunding -> refunded
                              -> refund_failed
pending_review -> rejected
```

说明：

1. `pending_review`：规则无法自动确认，或需要运营审核。
2. `approved`：审核通过，等待执行退款。
3. `refunding`：已调用退款渠道，等待回调。
4. `refunded`：退款完成。
5. `refund_failed`：退款渠道返回失败或回调失败。
6. `rejected`：审核拒绝退款。

第一阶段如果暂不接微信退款，`approved -> refunding -> refunded` 可以由后台人工操作推进：运营审核通过后进入 `approved`，财务或运营在线下完成微信商户后台退款后，再手动标记为 `refunded`。第二阶段接入微信退款接口后，再由系统调用和回调自动推进 `refunding/refunded/refund_failed`。

订单上的 `refund_status` 只保存汇总状态，不承载完整审核过程：

```text
none
pending_review
refunding
part_refunded
refunded
refund_failed
refund_rejected
```

### 14.1 退款策略分流建议

退款策略服务可以先提供一个纯规则评估入口，输入订单快照、取消上下文和历史取消统计，输出退款建议，不直接改订单和退款单。

第一阶段建议输出以下动作：

| 动作 | 含义 |
| --- | --- |
| `NO_REFUND_REQUIRED` | 未支付订单关闭，无需退款 |
| `SUGGEST_FULL_REFUND` | 明确建议全额退款；当前标准场景可直接自动执行 |
| `SUGGEST_PARTIAL_REFUND` | 明确建议部分退款，按阶梯扣费，仍可等待后台审核确认 |
| `ENTER_REVIEW` | 复杂或缺信息场景，进入人工审核 |
| `SUGGEST_REJECT` | 明确不退款场景，建议拒绝退款，仍需保留审核记录 |

建议第一阶段策略顺序：

1. 未支付订单：返回 `NO_REFUND_REQUIRED`。
2. 已支付但缺少 `planned_start_at`：返回 `ENTER_REVIEW`。
3. 已支付且服务已开始或存在现场爽约证据：返回 `ENTER_REVIEW`，不自动拒绝。
4. 已支付且包含特殊优惠权益：返回 `ENTER_REVIEW`。
5. 已支付且当月用户原因取消超过 3 次：返回 `ENTER_REVIEW`，等待运营确认频繁取消规则是否覆盖其他规则。
6. 已支付且距离预约开始超过 24 小时，且未命中频繁取消或特殊优惠：返回 `SUGGEST_FULL_REFUND`，并可直接自动执行退款。
7. 已支付且距离预约开始 12-24 小时，且为用户原因取消：返回 `SUGGEST_PARTIAL_REFUND`，服务费率 3%。
8. 已支付且距离预约开始 2-12 小时，且为用户原因取消：返回 `SUGGEST_PARTIAL_REFUND`，服务费率 5%。
9. 已支付且距离预约开始 2 小时内，且为用户原因取消：返回 `SUGGEST_PARTIAL_REFUND`，服务费率 8%。

当前实现中，`SUGGEST_FULL_REFUND` 在标准低风险场景下已经允许直接进入自动退款；其他策略仍只负责给出建议和原因，不负责最终审核结论。
