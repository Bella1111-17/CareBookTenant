# 微信支付 V3 接入设计

## 目标

本文面向当前 `CareBook` 服务端项目，目标是在 `apps/server/src/module/wechat` 下实现一个可复用、低耦合的微信支付 V3 能力，并与 `apps/server/src/module/care/care-order` 做业务关联，优先支持微信小程序支付。

本文不直接把支付逻辑写死在 `care-order` 中，而是先明确分层、配置、密钥管理和接口边界，保证后续可以平滑扩展到退款、关闭订单、支付查询，甚至支付宝等其他支付渠道。

## 结论先说

### 0. 退款结果不能只靠“申请退款接口返回成功”

微信退款接口返回 `200 OK` 只能说明微信已受理退款申请，不代表资金已经退回。

当前项目对退款结果的正确处理方式应当是：

- 退款申请接口负责把本地退款单推进到 `refunding`
- 微信退款回调作为主驱动，接到 `REFUND.SUCCESS` / `REFUND.CLOSED` / `REFUND.ABNORMAL` 后更新本地状态
- 后端增加退款查询能力，定时扫描仍处于 `refunding` 的退款单并向微信补查

不建议只做“退款发起 10 秒后查一次”这种一次性补偿。原因很直接：

- 微信退款到账时间不一定稳定
- 回调可能延迟，但不是永久丢失
- 一次查询命中不了就还会留下脏状态

所以这里采用：

- 回调优先
- 定时补偿查询兜底

### 1. 支付模块要独立于 `care-order`

建议拆成两层：

- `wechat` 模块：负责微信开放平台相关能力，包括现有 `code2Session`、`access_token`，以及新增的 `pay` 能力。
- `payment` 抽象层：定义项目内部统一的支付接口，不直接暴露微信 SDK 细节。

`care-order` 只调用项目内部支付接口，不直接关心签名、证书、回调验签、平台证书解密等微信支付细节。

### 2. 优先支持小程序支付场景

当前业务应走微信支付 V3 的 JSAPI 下单能力，小程序端拿到后端返回的支付参数后，调用 `wx.requestPayment` 发起支付。

服务端核心职责：

- 创建业务订单
- 创建支付单
- 调用微信支付统一下单
- 返回小程序拉起支付所需参数
- 接收微信支付回调
- 验签、解密、幂等处理
- 更新支付单和业务订单状态

### 3. 两个关键密钥不要直接写进 `dev.yml`

如果你说的“两个密钥”指的是下面这两个：

- 商户 API 私钥
- 微信支付平台证书公钥或平台证书文件

不建议把它们的完整内容直接放在 `apps/server/src/config/dev.yml`。

更合适的方式是：

- `yml` 里只放路径、商户号、序列号、回调地址等短配置
- 私钥和平台证书放到服务器本地受控目录，以文件形式读取

原因：

- PEM 内容很长，放进 YAML 可读性很差
- 容易误提交到 Git
- 多环境切换困难
- 后续轮换平台证书或商户私钥时不方便

## 推荐目录结构

建议在现有 `apps/server/src/module/wechat` 基础上扩展，不要把所有支付代码塞进一个 service。

推荐结构：

```text
apps/server/src/module/wechat/
├─ wechat.module.ts
├─ wechat.service.ts
├─ pay/
│  ├─ wechat-pay.module.ts
│  ├─ wechat-pay.service.ts
│  ├─ wechat-pay.constants.ts
│  ├─ wechat-pay.types.ts
│  ├─ wechat-pay.helper.ts
│  ├─ dto/
│  │  ├─ create-jsapi-order.dto.ts
│  │  └─ wechat-pay-notify.dto.ts
│  └─ interfaces/
│     └─ payment-provider.interface.ts
```

如果你后面准备把支付抽象为更通用能力，也可以再补一层：

```text
apps/server/src/module/payment/
├─ payment.module.ts
├─ payment.service.ts
├─ payment.types.ts
└─ interfaces/
   └─ payment-provider.interface.ts
```

这种分法下：

- `wechat/pay` 负责微信支付协议细节
- `payment` 负责统一业务接口
- `care-order` 只依赖 `payment`

## 为什么不建议优先找 “Nest 专用微信支付包”

我对这个场景的建议是：

- 可以参考成熟 Node 包的设计
- 但不要把项目核心能力押在一个薄封装的 Nest 三方模块上

原因：

- 很多 Nest 版微信支付包本质上只是再包一层 Node SDK
- 维护频率通常不稳定
- 回调验签、平台证书轮换、异常处理不一定适合你现在的项目
- 一旦包作者停更，后续升级成本反而更高

更稳妥的做法：

- 业务层自己定义接口
- 协议层可以选成熟 Node 包，也可以自己手写
- 即使底层实现换包，`care-order` 也不用改

## 建议的数据分层

不要把所有支付字段都堆进 `care_order` 表。

建议新增支付单表，至少把业务订单和支付单拆开。

推荐关系：

- `care_order`：护理业务订单
- `payment_order`：支付单，记录支付渠道、微信预支付单号、微信交易号、回调结果等

- `serviceItemId`
- `amount`
- `orderStatus`
- `serviceDate`

### `payment_order` 建议新增字段

- `id`
- `businessType`
- `businessOrderId`
- `businessOrderNo`
- `payChannel`
- `payScene`
- `merchantOrderNo`
- `wechatPrepayId`
- `wechatTransactionId`

- `amount`

- `payStatus`
- `notifyStatus`

`payment_order.payStatus`：

- `created`
- `paying`
- `paid`
- `closed`
- `failed`
- `refunded`

`care_order.orderStatus` 可以继续保留你现在的业务状态：

- `pending_payment`
- `pending_dispatch`
- `pending_service`
- `in_service`
- `completed`
- `cancelled`

支付成功后，再由支付回调驱动 `care_order` 从 `pending_payment` 更新到 `pending_dispatch`。

## 小程序支付完整流程

### 1. 小程序创建护理订单

小程序调用：

- `POST /care/client/order`

后端创建 `care_order`，初始状态：

- `pending_payment`

### 2. 小程序发起支付

建议新增接口：

- `POST /care/client/order/:id/pay`

后端处理流程：

1. 校验订单存在且属于当前用户
2. 校验订单状态必须是 `pending_payment`
3. 根据当前登录用户拿到 `openid`
4. 创建或复用一条未完成 `payment_order`
5. 调用微信支付 V3 JSAPI 下单接口
6. 保存 `prepay_id`
7. 生成小程序 `wx.requestPayment` 所需参数并返回

### 3. 小程序拉起微信支付

后端返回给小程序的内容通常包括：

- `timeStamp`
- `nonceStr`
- `package`
- `signType`
- `paySign`

其中：

- `package` 形如 `prepay_id=xxx`
- `signType` 通常使用 `RSA`

### 4. 微信服务端回调支付结果

建议新增回调接口：

- `POST /wechat/pay/notify`

处理流程：

1. 读取请求头中的微信签名相关字段
2. 验证回调签名
3. 解密 `resource.ciphertext`
4. 读取回调中的商户订单号、交易号、支付结果
5. 幂等处理
6. 更新 `payment_order`
7. 更新 `care_order.orderStatus`
8. 返回微信要求的成功响应

### 5. 订单状态变更

建议规则：

- 微信支付成功：`care_order.pending_payment -> pending_dispatch`
- 微信支付失败或关闭：保持或更新为业务可识别状态
- 用户取消支付不直接改业务单，只有明确关闭支付单后再处理

## 退款链路设计

### 1. 退款发起阶段

商户调用 `POST /v3/refund/domestic/refunds` 成功后，只表示微信已受理退款。

此时本地应做的事情：

- `payment_refund_order.refundStatus = refunding`
- 记录 `wechatRefundId`
- 业务订单 `care_online_order.refundStatus = refunding`
- 保持支付状态仍为 `paid`

这里不要直接把本地退款单改成 `refunded`。

当前项目在退款发起阶段增加了一条保守自动化规则：

- 已支付
- 服务未开始
- 距离预约开始时间 `> 24 小时`

命中以上条件时：

- 退款申请单可自动进入 `approved`
- 系统立即发起微信退款
- 业务订单进入 `refunding`

其他场景仍保留人工审核入口。

### 2. 退款回调阶段

微信退款结果回调仍会打到统一回调入口：

- `POST /wechat/pay/notify`

注意这里的关键点不是解密后的资源对象，而是回调报文顶层明文 JSON 中的 `event_type`。

退款回调的顶层事件通常包括：

- `REFUND.SUCCESS`
- `REFUND.CLOSED`
- `REFUND.ABNORMAL`

当前实现里必须先从 `rawBody` 解析顶层 `event_type`，再决定走支付回调还是退款回调分支，不能只依赖 `@Body().event_type`。

原因：

- 某些运行链路下 `@Body()` 取值不稳定
- 一旦分流失败，退款通知会被错误当成支付通知处理
- 本地退款单就会一直卡在 `refunding`

### 3. 退款状态映射

建议使用下面的映射规则：

微信退款状态 -> 本地退款单状态

- `PROCESSING` -> `refunding`
- `SUCCESS` -> `refunded`
- `CLOSED` -> `refund_failed`
- `ABNORMAL` -> `refund_failed`

业务订单状态联动：

- 全额退款成功：`care_online_order.refundStatus = refunded`，`paymentStatus = refunded`
- 部分退款成功：`care_online_order.refundStatus = part_refunded`，`paymentStatus = part_refunded`
- 退款关闭或异常：`care_online_order.refundStatus = refund_failed`，`paymentStatus` 保持 `paid`

### 4. 退款查询补偿

微信官方明确要求商户不能只依赖回调通知获取退款结果，需要结合查询接口做兜底。

因此服务端需要补上：

- 查询接口：`GET /v3/refund/domestic/refunds/{out_refund_no}`

建议补偿策略：

- 使用 `monitor/job` 统一任务体系，不在业务 Service 内直接挂 `@Cron`
- 通过 `task.reconcileWechatRefundStatus` 这类任务入口调业务 Service
- 由 `sys_job` 配置实际执行频率
- 只处理 `refundStatus = refunding` 的记录
- 只补查最近仍未终态、且最近 30 秒未更新的退款单
- 每轮限制批量数，避免瞬时打满微信接口

当前实现默认建议频率：

- 每 10 分钟一次全局补偿查询
- 如需加快状态收敛，可后续再叠加一次短延迟补查，但不替代全局兜底

这个策略比“退款后固定 10 秒查一次”更稳：

- 回调正常时不会额外频繁查单
- 回调漏发、分流失败、网络抖动时可以自动纠偏
- 对运营来说，本地状态会自然收敛到真实状态

### 5. notify_url 约束

退款申请时如果请求体显式传了 `notify_url`，则该地址优先于商户平台的退款通知配置。

当前项目建议继续在退款申请里显式传统一通知地址：

- `https://xxx/prod-api/wechat/pay/notify`

这样可以避免：

- 商户平台配置和代码配置不一致
- 支付回调、退款回调落到不同入口

### 6. 回调处理原则

回调处理必须满足以下原则：

- 先验签，再解密
- 先快速应答，再做业务处理
- 业务处理必须幂等
- 重复回调必须可重复消费
- 回调之外仍保留查单兜底

## 与 `care-order` 的解耦方式

`care-order` 不应该直接写这样的逻辑：

- 拼微信支付请求体
- 用私钥签名
- 验证微信回调签名
- 解析平台证书

这些都应该留在 `wechat/pay` 模块内。

### 推荐依赖方向

```text
care-order -> payment abstraction -> wechat pay provider
```

### 推荐职责划分

`CareOrderService`

- 创建业务订单
- 校验业务归属
- 发起支付请求
- 在支付成功后推进业务状态

`WechatPayService`

- 创建 JSAPI 预支付单
- 生成小程序调起支付参数
- 查询微信订单
- 关闭微信订单
- 验签和解密回调

`PaymentService`

- 创建支付单
- 管理支付状态
- 协调业务订单与支付单关系
- 调用底层 `WechatPayService`

## 配置设计建议

当前项目已有：

```yml
wechat:
  appId: '...'
  appSecret: '...'
```

建议新增一个独立的支付配置段，不要把开放平台登录配置和支付配置混在一起。

推荐配置：

```yml
wechat:
  appId: '小程序 appid'
  appSecret: '小程序 secret'
  pay:
    enabled: true
    mchId: '商户号'
    mchSerialNo: '商户证书序列号'
    apiV3Key: 'APIv3 密钥'
    notifyUrl: 'https://admin.xxx.com/prod-api/wechat/pay/notify'
    privateKeyPath: './secrets/wechat/apiclient_key.pem'
    platformCertPath: './secrets/wechat/wechatpay_platform.pem'
```

这里建议区分清楚几个概念：

- `appId`：小程序 appid
- `mchId`：微信支付商户号
- `mchSerialNo`：商户 API 证书序列号
- `apiV3Key`：商户平台设置的 APIv3 密钥
- `privateKeyPath`：商户 API 私钥文件路径
- `platformCertPath`：微信支付平台证书文件路径

## 那两个密钥放哪里最合适

如果按你当前项目习惯继续使用 `apps/server/src/config/dev.yml`，建议这样处理：

### 可以放进 `dev.yml` 的

- `mchId`
- `mchSerialNo`
- `notifyUrl`
- `apiV3Key`
- `privateKeyPath`
- `platformCertPath`

### 不建议直接放进 `dev.yml` 的

- 商户私钥 PEM 内容
- 微信支付平台证书 PEM 内容

## 小程序支付所需的用户标识

JSAPI 小程序支付必须有 `openid`。

所以当前项目还需要确认一件事：

- 登录态里是否已经把当前用户对应的微信 `openid` 保存到数据库

如果没有，后续必须补齐这条链路：

1. 小程序 `wx.login`
2. 后端 `code2Session`
3. 拿到 `openid`
4. 将 `openid` 和系统用户绑定
5. 发起支付时按当前用户取 `openid`

如果没有用户 `openid`，就没法走小程序 JSAPI 支付。

## 回调接口的注意点

### 1. 不要依赖前端回传支付成功

最终支付结果必须以后端回调为准，不要以前端 `requestPayment success` 为最终成功依据。

### 2. 必须做幂等

微信回调可能重复推送，回调处理必须幂等。

建议用：

- `merchantOrderNo`
- `wechatTransactionId`

做唯一约束或幂等检查。

### 3. 验签失败直接拒绝

不能只解密不验签，也不能因为调试方便就跳过验签。

### 4. 金额必须二次校验

回调成功后，需要校验：

- 商户订单号是否匹配
- 金额是否匹配
- 支付状态是否是成功

金额不一致不能直接把业务单改成已支付。

## 订单号设计建议

线上订单号建议由后端统一生成纯数字可读单号，前端只原样展示，不自行裁剪前缀。

当前线上订单已采用“时间戳 + 随机段”的纯数字方案，例如：

```text
20260710153045123123
```

设计要求：

- 纯数字，便于用户口述和客服检索
- 大体按时间递增，便于人工排查
- 不在 ToC 订单号中暴露 `OL` / `CO` 等类型前缀

你之前 `care-order` 的订单号生成方式是：

```ts
private buildOrderNo() {
  return `CO${Date.now()}`
}
```

这个方式在并发下不够稳妥，建议后续改造：

- 增加随机段
- 或引入雪花 ID
- 或单独做订单号生成器

支付单号建议单独生成，不复用业务订单号本体。

例如：

- 业务订单号：`20260710153045123123`
- 支付单号：`CP20260710153045123123`

## 现阶段最关键的前置条件

- 微信支付商户号 `mchId`
- 商户 API 证书序列号 `mchSerialNo`
- APIv3 密钥 `apiV3Key`
- 商户 API 私钥文件 `apiclient_key.pem`
- 微信支付平台证书文件
- 支付回调公网地址 `notifyUrl`
- 当前用户与微信 `openid` 的绑定方式

## 对你当前问题的直接回答

### 文档先写在哪

建议就放在：

- `docs/integrations/微信支付V3接入设计.md`

### 那两个密钥放哪里合适

如果是支付相关“两个密钥/证书”：

- `dev.yml` 里放路径，不放 PEM 正文
- 本地开发放 `apps/server/secrets/wechat/`
- 生产环境放部署机安全目录

### `dev.yml` 里应该放什么

放这些短配置最合适：

```yml
wechat:
  appId: '...'
  appSecret: '...'
  pay:
    enabled: true
    mchId: '...'
    mchSerialNo: '...'
    apiV3Key: '...'
    notifyUrl: 'https://admin.xxx.com/prod-api/wechat/pay/notify'
    privateKeyPath: './secrets/wechat/apiclient_key.pem'
    platformCertPath: './secrets/wechat/wechatpay_platform.pem'
```
