# CareBook 产品需求文档（PRD）

本文档是 CareBook 护理业务的产品需求主入口，定义订单、履约、服务证据、护理日报、通知触达和日报问答的业务范围。具体数据库设计、API、组件实现、定时任务配置和 Prompt 模板放在 `docs/engineering/`、`docs/integrations/` 和 `docs/operations/` 中维护。

## 1. Executive Summary

### Problem Statement

CareBook 同时服务 ToC 家属用户和 ToB 机构/渠道订单。当前产品的核心问题不是单纯创建订单，而是要把“护理服务确实发生了、服务过程可追溯、家属能理解服务结果”做成稳定闭环。

护理现场存在音频嘈杂、设备绑定不稳定、线下订单多、同日改派和服务证据缺失等复杂情况。如果没有统一履约主轴和明确的低置信度处理策略，护理日报容易出现漏生成、错归属、事实不准或无法解释的问题。

### Proposed Solution

构建统一的护理履约平台，让 ToC 在线订单和 ToB/线下订单共享同一条业务主轴：

```text
订单/线下服务来源 -> 被护理人 -> 指派护工 -> 服务关系 -> 护工绑定设备证据路由 -> AI 处理 -> 正式护理日报 -> 微信通知 -> 小程序查看/问答
```

平台用“服务关系”作为履约事实的中心对象，用智能工牌或等效证据支撑护理日报生成，用低置信度、草稿、待审核机制处理证据不足场景，并通过小程序向家属交付可信、可理解的日报结果。

### Success Criteria

当前阶段不以经营 KPI 或复杂数据指标作为 PRD 的重点。产品需求先以以下状态作为成功判断：

- ToC 在线订单和 ToB/线下订单能进入同一套履约、日报和通知流程。
- 平台运营人员能清楚地建立、调整和追溯服务关系。
- 家属能围绕被护理人查看一份面向人的正式护理日报，而不是设备流水或后台分析结果。
- 证据不足、设备缺失、低置信度和同日改派等复杂场景有明确降级路径。
- 聊天机器人只能基于护理日报事实回答问题，不能把推测当作服务事实。

## 2. User Experience & Functionality

### User Personas

| 角色 | 主要目标 | 关键关注点 |
| --- | --- | --- |
| ToC 家属用户 | 下单、绑定被护理人、查看每日护理结果 | 报告是否可信、是否及时、有无异常 |
| ToB 机构/渠道运营商 | 创建和管理机构/保司/线下护理订单 | 标准化履约、服务反馈、客户交付 |
| 平台运营人员 | 录入订单、派单、改派、处理异常 | 操作效率、数据完整性、履约连续性 |
| 护工 | 接收指派并完成护理服务 | 工作内容被准确记录，不被错误归属 |
| 质检经理 | 审核服务质量和 AI 日报质量 | 可追溯、可抽检、可干预 |
| 家属问答用户 | 用自然语言询问护理情况 | 回答必须基于日报事实，不臆造 |

### Core User Flows

#### Flow A：ToC 在线订单

1. 家属用户进入小程序。
2. 家属创建或选择被护理人档案。
3. 家属选择护理服务并提交订单。
4. 平台完成支付/订单确认流程；取消与退款口径见 `docs/product/服务退款规则.md`。
5. 平台运营人员指派已绑定智能工牌的护工。
6. 系统建立服务关系并开始履约。
7. 智能工牌或等效证据系统上传服务证据。
8. 次日 7:00 定时任务生成前一日正式护理日报。
9. 系统发送微信轻量通知。
10. 家属进入小程序查看日报，并可基于日报进行问答。

#### Flow B：ToB/线下订单

1. 平台运营人员获得机构、保司或线下渠道服务需求。
2. 运营人员录入被护理人信息、联系人和服务信息。
3. 运营人员创建线下订单或无订单服务来源。
4. 运营人员指派已绑定智能工牌的护工。
5. 系统建立服务关系并执行同一套履约和日报生成流程。
6. 后续通过微信生态触达家属，家属认证后查看完整日报。

#### Flow C：同日改派

1. 运营人员结束原服务关系。
2. 系统记录原服务关系的结束时间、原因和操作人。
3. 运营人员创建新的服务关系。
4. 当天不创建正式护理日报，也不向家属推送通知。
5. 服务关系时间线必须保留交接或改派事实，供运营处理。

#### Flow D：护工开始与结束服务（MVP）

1. 已派单护工使用统一微信小程序进入护工工作台。
2. 系统仅展示当前登录护工本人处于 `pending` 或 `serving` 的服务关系。
3. 护工主动点击“开始服务”，服务关系转为 `serving`，订单转为“服务中”。
4. 护工可主动点击“结束服务”；已经开始且达到服务关系计划结束时间的服务，也由定时任务自动结束。
5. 未开始的已派单服务不得因计划时间到达而自动完成。

### User Stories & Acceptance Criteria

#### Story 1：被护理人和订单闭环

As a ToC 家属用户, I want to 为指定被护理人购买或接受护理服务, so that 我能远程了解老人是否得到实际照护。

Acceptance Criteria:

- 系统必须支持家属创建、编辑、选择自己名下的被护理人档案。
- 当前版本中，一个订单必须且只能绑定一个被护理人。
- 小程序必须展示被护理人的当前服务状态和历史日报列表。
- 家属只能查看当前归属于自己的被护理人及其日报。

#### Story 2：ToB/线下订单录入

As a 平台运营人员, I want to 录入机构、保司或线下来源订单, so that 这些订单不用走 ToC 前台流程也能进入统一履约闭环。

Acceptance Criteria:

- 管理后台必须支持录入线下订单、被护理人、联系人、服务地址和服务时间。
- ToB/线下订单必须进入与 ToC 订单相同的服务关系和日报生成管道。
- 线下订单不设置独立“已派单”状态，派单完成后统一进入“待服务”或“服务中”口径。
- 历史订单必须保留服务快照和被护理人快照，避免后续档案变更影响历史记录。

#### Story 3：统一履约模型

As a 平台运营人员, I want to 用同一套履约模型管理 ToC 和 ToB 服务, so that 派单、改派、监管和日报生成保持一致。

Acceptance Criteria:

- 系统必须用服务关系表达“哪位护工服务哪位被护理人”。
- 服务关系必须支持订单来源和线下来源。
- 同一被护理人同一时刻只能存在一条 `serving` 状态服务关系。
- 同一订单同一时刻只能存在一条有效履约关系。
- 改派必须走“关旧开新”的原子过程，并保留操作日志。

#### Story 4：护工与设备强绑定

As a 平台运营人员, I want to 指派已绑定智能工牌的护工, so that 系统能正确追溯服务证据来源。

Acceptance Criteria:

- 派单时必须指定护工，不能直接指派设备。
- 可被派单的护工必须同时具有 `caregiver` 角色、在用护工扩展档案和有效智能工牌绑定。
- 每名护工和每台智能工牌在同一时刻均只能存在一条有效绑定关系。
- 管理后台只提供护工选择；没有有效设备绑定的护工不可被指派。
- 日报必须通过护工在服务时间段内的设备绑定历史关联证据，不能使用“当前设备”替代历史绑定。

#### Story 5：正式护理日报生成

As a ToC 家属用户, I want to 每天收到一份清晰、可信的护理日报, so that 我无需阅读原始记录也能了解当天服务情况。

Acceptance Criteria:

- 系统默认在每日 7:00 扫描前一日活跃服务并生成日报。
- 正式日报必须按“同一订单 + 同一自然日”去重，同一订单同一天最多生成一份正式日报。
- 日报必须包含被护理人、报告日期、服务摘要、护工来源、任务完成亮点、异常提醒和置信度状态。
- 同日同订单发生不同护工改派时，不生成正式日报、不推送家属通知；不得自动合并多名护工的证据。
- 系统不得把原始嘈杂 transcript 直接展示给家属。

#### Story 6：证据不足和低置信度处理

As a 质检经理, I want to 识别证据缺失、低置信度和可疑日报, so that 平台不会把不可靠内容当成确定事实推送给家属。

Acceptance Criteria:

- 护工没有有效设备绑定、无法命中唯一设备或没有可用证据时，系统不得生成正式日报，并暂停推送。
- AI 处理结果低置信度时，系统必须标记原因，不得臆造确定性服务细节。
- 运营后台必须能查看低置信度、草稿、待审核和推送失败的日报清单。
- 每份正式日报必须保留从报告到服务关系、护工/设备、AI 证据的追溯路径。

#### Story 7：护工工作准确反映

As a 护工, I want to 我的实际服务被准确归属和总结, so that 平台、机构和家属看到的日报不会错误归因。

Acceptance Criteria:

- 日报不得将其他护工、其他设备或其他被护理人的服务内容错误归属到当前护工。
- 护工改派日必须按服务关系时间段保留证据；当前版本不生成该订单当日的家属日报。
- 如果音频证据无法可靠判断角色或动作，系统必须标记低置信度或进入人工审核。

#### Story 8：微信通知和报告查看

As a 家属用户, I want to 收到轻量通知并进入小程序查看完整日报, so that 我能及时了解服务情况且隐私不泄露在通知内容中。

Acceptance Criteria:

- 通知内容只承载跳转和必要提示，不包含完整护理日报正文。
- 用户进入小程序后必须通过认证和被护理人归属校验才能查看报告。
- 通知发送失败必须记录失败原因，并进入运营可查看的失败清单。
- 通知必须在日报生成完成后触发，不得在草稿或待审核状态下发送。

#### Story 9：护理日报问答

As a 家属用户, I want to 通过聊天机器人询问当天护理情况, so that 我能快速获得关键信息而不必逐段阅读日报。

Acceptance Criteria:

- 机器人必须能回答“今天做了什么”“服务情况如何”“有没有异常”等护理日报相关问题。
- 机器人回复必须基于护理日报、服务关系、护工和被护理人结构化上下文。
- 上下文优先携带最近日期日报，默认向前追溯不超过 7 天。
- 当日报不存在或数据不足时，机器人必须明确说明暂无可用报告，不得编造内容。
- 机器人回复中的事实性内容必须可追溯到日报或相关结构化数据。

#### Story 10：护工最小履约操作

As a 护工, I want to 在本人小程序工作台开始或结束被派发的服务, so that 服务状态能反映实际履约进展。

Acceptance Criteria:

- 护工工作台只能展示当前登录护工本人被派发且未结束的服务关系。
- 护工工作台及其菜单入口均由现有 `caregiver` RBAC 角色控制；不匹配角色不展示护理员菜单，不新增角色或账号体系。
- 护工角色仅决定菜单和接口入口；具体任务仍须校验服务关系归属，护理员不能操作其他护理员的任务。
- `pending` 服务仅可由该护工开始；开始后记录服务端时间并转为 `serving`。
- `serving` 服务仅可由该护工结束；结束后记录服务端时间并转为 `completed`。
- 已开始且计划结束时间已到的服务由统一任务自动结束；未开始的 `pending` 服务不得自动结束。
- 开始、结束和自动结束必须同步对应线上或线下订单的服务状态。
- 当前版本不采集位置、照片、人脸或工牌近场证明；不创建独立的每日签到记录。

### Non-Goals

1. 多家属成员、共同照护和细粒度家庭权限模型不在 v1.0 范围内。
2. 医生端、临床诊疗端和医疗建议功能不在本 PRD 范围内。
3. 第三方机构多租户产品化不在第一版正式发布范围内，仅保留未来扩展路径。
4. 高级 BI、机构分析仪表盘、深度质量评分和趋势预测不在 v1.0 范围内。
5. 邮件、富媒体短信等跨渠道报告推送不在 v1.0 范围内。
6. 复杂支付扩展、分账、组合支付和自动化退款编排不在本 PRD 主体内；取消与退款产品口径单独维护在 `docs/product/服务退款规则.md`。
7. 原始音频或完整 transcript 面向家属开放查看不在 v1.0 范围内。
8. 护工档案、微信邀请绑定、角色切换、多次上门服务记录、定位与反作弊签到不在本次 MVP 范围内，后续以 `CaregiverProfile` 和 `CareServiceSession` 分别承载。

## 3. AI System Requirements

### Tool Requirements

#### 智能工牌或等效证据采集

- 必须采集护理服务期间的音频或等效服务证据。
- 必须把证据与设备号、服务时间和服务关系建立可追溯关联。
- 必须支持证据缺失、上传延迟和设备未绑定场景的异常记录。

#### ASR 与 transcript 处理

- 必须支持普通话护理场景语音转文本。
- 当前可使用 `Fun-ASR` 或等效 ASR 服务，生产使用前必须完成验证。
- 必须清洗无效噪声、重复片段、无关语音和明显不可用内容。
- 必须对低置信度片段分类，而不是静默当作可靠事实。

#### 角色归属与事件提取

- 系统应在 transcript 允许可靠推断时区分护工行为、被护理人状态和环境信息。
- 必须提取日报所需结构化事实：服务活动、被护理人状态、异常事件、家属关注事项。
- 对无法可靠归属的内容必须保留不确定性标记。

#### 摘要生成

- 必须生成面向家属的 plain-language 摘要。
- 摘要不得加入 transcript 或结构化事实中不存在的服务细节。
- 摘要必须避免把猜测、建议或医学判断包装成确定事实。

#### 人工审核钩子

- 低置信度、证据不足、异常事件和改派复杂场景必须支持人工审核。
- 审核结果必须可回写日报状态，并保留审核人、审核时间和处理意见。

#### 日报问答上下文

- 聊天机器人必须自动携带被护理人相关日报上下文。
- 上下文数据源包括护理日报、服务关系、护工资料、被护理人信息和日报置信度。
- 当上下文不足时，机器人必须显式降级为“暂无足够报告信息”。

### Evaluation Strategy

#### 样本集

使用真实或隐私安全处理后的护理录音、transcript 和日报样本验证 AI 流程，至少覆盖：

- 安静环境服务。
- 嘈杂环境服务。
- 断续或多人说话场景。
- 护工改派日。
- 设备缺失或绑定异常。
- 被护理人异常事件。
- 无有效证据但存在订单/服务关系的场景。

#### AI 质量原则

- Transcript 应能支撑服务事实判断，无法支撑时必须进入低置信度或待审核状态。
- 角色归属不清时，系统不能把不确定内容归到具体护工或被护理人身上。
- 结构化事实提取应优先保留护理动作、被护理人状态、异常事件和家属关注事项。
- 摘要必须和底层事实一致，不加入证据中不存在的服务细节。
- 证据不足是正常业务状态，系统必须显式降级，而不是生成看似完整但无法解释的报告。

#### 问答质量原则

- 机器人回复必须能追溯到护理日报或相关结构化数据。
- 没有日报、日报状态不适合回答或上下文不足时，机器人必须明确说明信息不足。
- 机器人不能提供医疗诊断、治疗建议或日报之外的推断性结论。
- 机器人应优先用家属能理解的表达回答，避免暴露内部字段、Prompt 或原始 transcript。

#### 人工复核

- 产品上线初期应保留人工复核机制，重点检查事实正确性、角色正确性、可读性和低置信度标记是否合理。
- 抽检结果应反馈到规则、Prompt、模型或人工审核策略中。

## 4. Technical Specifications

### Architecture Overview

第一版正式发布采用统一服务生命周期：

```text
订单录入
  -> 履约设置
  -> 服务证据采集
  -> AI 处理
  -> 正式日报组装
  -> 通知与查看
  -> 监管与修正
```

#### 订单录入

- ToC 用户通过小程序创建订单。
- ToB/线下订单通过管理后台或运营流程录入。
- 每个订单最终只对应一个被护理人。

#### 履约设置

- 平台通过服务关系承载履约事实。
- 派单必须指派已绑定智能工牌的护工，设备不作为履约主体。
- 改派必须关闭旧服务关系并开启新服务关系。

#### 服务证据采集

- 智能工牌或等效证据系统上传音频、设备号、时间和相关分析结果。
- 系统通过服务关系确定证据属于哪个订单、被护理人和履约段。

#### AI 处理

- ASR 输出经过清洗、角色归属、事实提取和置信度判断。
- 系统生成面向日报的结构化证据层，不把原始 transcript 直接发送给家属。

#### 正式日报组装

- 系统按订单和自然日生成正式护理日报。
- 同日多段服务关系合并为一份日报。
- 低置信度或证据不足进入草稿/待审核状态。

#### 通知与查看

- 正式日报生成后触发微信通知。
- 家属在小程序中完成身份和被护理人归属校验后查看完整报告。

#### 监管与修正

- 运营后台展示生成失败、证据缺失、低置信度、待审核和推送失败清单。
- 质检人员可查看证据链、审核日报和记录处理结果。

### Security & Privacy

- 音频、transcript、日报摘要、被护理人档案和服务记录必须按敏感个人数据处理。
- 通知 payload 只包含进入认证查看流程所需的最小信息。
- 家属查看权限必须基于被护理人当前归属判断。
- 被护理人归属变更必须记录日志，线下反向激活允许覆盖当前归属但必须可审计。
- 运营、护工、机构和家属之间必须保持访问控制边界。
- 原始证据默认不面向家属展示，只供质量审核和问题追溯使用。
- 数据保留、删除策略、跨境合规要求为 `TBD`，生产规模化前必须补齐。

### Testing Requirements

- 后端必须覆盖护工有效设备绑定校验、同一被护理人 serving 唯一性、订单日报去重、同日改派不出日报和低置信度降级逻辑。
- 小程序必须覆盖被护理人选择、日报列表、日报详情、无日报状态和问答入口的关键路径。
- 管理后台必须覆盖线下订单录入、已绑定设备护工派单、改派、待审核日报和推送失败清单。
- AI 流程必须有离线评估集和每周生产抽检机制。
- 通知流程必须覆盖发送成功、发送失败、草稿不发送和权限校验失败场景。

## 5. Risks & Roadmap

### Phased Rollout

#### MVP（当前可运行基线）

- 基础 ToC/ToB 订单流程可用。
- 核心服务关系和日报生成路径已存在。
- 初步智能工牌或等效证据链路已接入。
- 部分异常处理和运营干预路径需要补强。

#### v1.0（第一版正式发布）

- 统一 ToC 和 ToB 履约主轴。
- 强化派单双轨互斥和服务关系唯一性。
- 提升正式护理日报生成可靠性和可衡量性。
- 建立证据不足、低置信度、草稿和待审核处理机制。
- 标准化微信通知和小程序报告查看。
- 上线基于日报上下文的聊天机器人问答。
- 建立每周 AI 日报质量抽检和运营异常 review 机制。

#### v1.1

- 增强日报审核工作流。
- 增加缺失证据、低置信度、护工异常模式和推送失败的质量看板。
- 基于真实生产样本持续优化 ASR、规则、Prompt 和模型策略。
- 扩展更多机构字段和渠道运营能力。

#### v2.0

- 评估第三方平台多租户架构。
- 扩展医院、养老院、保险渠道的合作伙伴能力。
- 添加高级质量评分、趋势分析和监管工具。
- 评估多家属协同查看和更细权限模型。

### Technical Risks

| 风险 | 描述 | 缓解措施 |
| --- | --- | --- |
| 音频噪声 | 环境嘈杂、多人说话、碎片语音降低 ASR 可靠性 | 噪声分类、低置信度标记、人工审核和备用报告路径 |
| 角色归属错误 | 护工行为、被护理人状态或环境声音被错误归类 | 用典型样本持续验证角色归属，无法可靠判断时降级处理 |
| 证据缺口 | 设备未绑定、上传失败、服务无音频导致无法生成可信报告 | 明确定义草稿/待审核/暂停推送状态 |
| 虚假置信度 | AI 生成流畅但事实错误的摘要 | 置信度门控、事实一致性评估和每周人工抽检 |
| 改派复杂性 | 同日多护工导致证据合并错误 | 服务关系“关旧开新”，当天不生成正式日报 |
| 通知闭环失败 | 通知发送失败或用户未打开导致日报触达不足 | 监控送达率、打开率和失败原因清单 |
| 权限误判 | 被护理人归属变化导致家属误看或看不到日报 | 当前归属校验、归属变更日志和权限测试 |
| 范围扩张 | 过早引入多租户、医生端、BI 等能力影响 v1.0 交付 | 用 Non-Goals 锁定范围，新增能力进入后续版本 |
