# 订单模块设计文档

## 1. 设计结论

订单模块采用全新的线上、线下双表设计。

1. `care_online_order`：线上订单，面向小程序 C 端用户，按"服务商品订单"设计。
2. `care_offline_order`：线下订单，面向后台运营录单、保险公司来源单、医院/机构合作单。
3. `care_recipient`：被护理人主档案，只保存人的稳定基础信息。
4. `care_relation`：服务关系，承接实际履约、护工绑定、上单/下单、改派历史。
5. `care_daily_report`：护理日报，按订单按天聚合生成，底层数据来自具体服务关系。

本设计不兼容旧的统一订单表思路，后续开发、接口和迁移都以 `care_online_order` 与 `care_offline_order` 为准。

核心边界：

1. 订单解决"怎么买服务、从哪里来、预约什么时候服务"。
2. 服务关系解决"哪位护工实际服务哪位被护理人"。
3. 护理日报对用户侧按订单按天聚合生成，底层履约明细仍追溯到服务关系。
4. 被护理人档案不承载订单级医院、科室、床位、病情、预约时间等快照信息。
5. 护工资料继续使用 `sys_user + caregiver_profile`，订单和服务关系只保存 `caregiver_user_id`。
6. 一个订单可以对应多条 `care_relation`，用于改派、顶班、分段服务。
7. 线下订单以运营录单表头为准，原始来源字段优先落快照，再由结构字段承接可检索内容。

本文档职责边界：

1. 本文档作为护理订单模块唯一主设计文档，统一维护订单域、服务关系、被护理人、服务商品、线下录单结构，以及正式护理日报的 V1 规则。
2. 其中订单表、服务关系、录单结构属于模型设计；护理日报的生成时机、主护工判定、跨天规则、回退规则、实施步骤也在本文档内统一收口。
3. 历史上的 `docs/engineering/护理订单履约日报V1实施方案.md` 已并入本文档，不再单独维护完整规则。

### 1.1 文档状态与阅读规则

本文档同时记录目标模型和迁移基线，但两者必须明确区分：

1. 除“当前实现基线”小节外，本文中的规则均为后续开发、重构和验收必须遵守的目标口径，不能根据现有表结构反向降低规则。
2. `护理订单履约日报V1实施方案.md` 只保留兼容跳转，不再新增规则、字段或实施步骤；所有调整只写回本文档。
3. 当前代码尚未完全达到本文目标。尤其是日报仍保留 relation 级唯一键、线上订单未进入订单级日报任务、服务关系的改派尚未具备事务与数据库唯一约束。相关差异在第 9.3 节集中记录。
4. 实施时先完成第 9.10 节数据模型收口和第 9.11 节迁移切片，再调整接口和页面；不得只在应用层绕过数据库约束。

通用技术约定：

1. 所有金额字段统一用整数分，例如 `30000` 表示 300 元；包括服务商品基准价、订单金额、退款金额和结算金额。
2. 预约时间、实际上单、实际下单等时间字段使用 PostgreSQL `timestamptz`。
3. 订单快照字段用于还原历史，不随服务商品、地址、被护理人档案后续修改而变化；线上订单同时保留商品快照和被护理人快照。
4. `banners`、`detail` 等多图字段在数据库中存为逗号分隔字符串，前端按需转为数组使用。

## 2. 线上订单：服务商品订单

### 2.1 表名

`care_online_order`

### 2.2 业务定位

线上订单本质是用户在小程序购买一个护理服务商品。服务管理里的 `care_service_item` 就是售卖对象，当前实体也已经定义为"护理服务商品"。

当前阶段采用"一单一服务商品"：

1. 不做购物车。
2. 不做多商品合并支付。
3. 不拆 `care_online_order_item`。
4. 订单主表保存 `service_item_id` 和商品快照。
5. 后续如果支持一次购买多个服务，再新增订单明细表。

### 2.3 与服务商品的关系

下单时从 `care_service_item` 读取：

1. `id`
2. `category_id`
3. `name`
4. `base_price_amount`
5. `duration_type`
6. `duration_value`
7. `channel_scope`
8. `banners`
9. `detail`

必须校验：

1. `status = '0'`，服务已上架。
2. `channel_scope in ('online', 'both')`，允许线上购买。

### 2.4 商品与被护理人双快照设计

线上订单使用 `service_item_snapshot(jsonb)` 保存商品展示快照，同时使用 `recipient_snapshot(jsonb)` 保存下单时的被护理人快照。关键订单字段必须拉平为订单列：

| 必须拉平的列        | 说明                 |
| ------------------- | -------------------- |
| `service_item_id`   | 服务商品 ID          |
| `quantity`          | 购买数量，默认 1     |
| `unit_price_amount` | 单价，单位：分       |
| `total_amount`      | 商品总金额，单位：分 |
| `payable_amount`    | 应付金额，单位：分   |
| `paid_amount`       | 实付金额，单位：分   |
| `planned_start_at`  | 预约开始时间         |
| `planned_end_at`    | 预约结束时间         |
| `order_status`      | 订单状态             |
| `payment_status`    | 支付状态             |
| `refund_status`     | 退款状态             |

适合放进 `service_item_snapshot` 的字段：

```json
{
  "id": 12,
  "categoryId": 2,
  "name": "住院24小时陪护",
  "basePriceAmount": 30000,
  "durationType": "day",
  "durationValue": 1,
  "channelScope": "online",
  "banners": "https://example.com/banner.png,https://example.com/banner2.png",
  "detail": "https://example.com/detail-1.png"
}
```

适合放进 `recipient_snapshot` 的字段：

```json
{
  "id": 25,
  "name": "王建国",
  "gender": "0",
  "birthday": "1946-03-02",
  "guardianName": "王芳",
  "guardianPhone": "13800000000",
  "illness": "术后恢复期"
}
```

说明：

1. `recipient_id` 继续保留为结构化外键。
2. `recipient_snapshot` 用于还原历史，不随被护理人主档案后续编辑而变化。
3. 线上、线下订单都遵循“主档案 + 快照”双轨策略，避免历史订单和历史日报被新数据污染。

### 2.6 订单状态

订单状态：

1. `created`：已创建，待支付/待发起支付。
2. `pending_dispatch`：已支付，待派单。
3. `pending_service`：已派单，待服务。
4. `in_service`：服务中。
5. `completed`：已完成。
6. `cancelled`：已取消。

支付状态（当前实现枚举）：

1. `unpaid` — 未支付
2. `paying` — 支付中
3. `paid` — 已支付
4. `closed` — 已关闭
5. `part_refunded` — 部分退款
6. `refunded` — 已退款
7. `pay_failed` — 支付失败

退款状态（entity 字段级别）：

1. `none` — 无退款
2. `pending_review` — 待审核
3. `refunding` — 微信退款处理中
4. `part_refunded` — 部分退款
5. `refunded` — 已退款
6. `refund_failed` — 退款失败
7. `refund_rejected` — 审核驳回

说明：

1. 中途终止且只退剩余未服务部分时，订单状态保持 `completed`，退款状态记录为 `part_refunded`。
2. 退款默认仍以运营审核为主，但标准低风险场景可开放自动退款；当前仅对“已支付、未开始服务、距离预约开始超过 24 小时”的线上订单开放自动全额退款。
3. 已提交微信退款后，优先依赖微信退款回调更新订单状态；如回调延迟或丢失，再由退款状态对账任务补偿查询。
4. 线上订单取消后的全额退款、部分退款、不退款条件见 `docs/product/服务退款规则.md`。

### 2.7 与服务关系的联动

1. 用户提交订单：`order_status = created`。
2. 支付成功：订单进入 `pending_dispatch`。
3. 后台派单：生成 `care_relation`，订单进入 `pending_service`。
4. 护工实际上单：对应服务关系进入 `serving`，订单同步展示为 `in_service`。
5. 当前服务关系结束：只代表一段履约结束，不代表整单完成。
6. 最后一段有效服务关系结束：订单才进入 `completed`。

订单状态用于 C 端展示大状态；服务关系状态用于内部履约明细。

## 3. 线下订单：机构/保险来源单

### 3.1 表名

`care_offline_order`

### 3.2 业务定位

线下订单用于承接：

1. 平安健康险等保险公司来源单。
2. 好医生等渠道来源单。
3. 医院、养老院、合作机构来源单。
4. 电话录单、后台人工录单。

线下订单的重点不是支付，而是来源、派单、服务安排、医院位置、核销和线下结算。

### 3.3 护工派单与设备绑定

订单和服务关系只指派护工，设备不参与派单。

1. 每条有效 `care_relation` 必须有 `caregiver_user_id`，不支持纯设备、外派设备或无护工履约。
2. `device_user_binding` 是护工与智能工牌的一对一绑定历史：同一时刻一名护工只能绑定一台设备，一台设备也只能绑定一名护工。
3. 派单前后端都必须校验护工存在当前有效设备绑定；未绑定设备的护工不能被指派。
4. 设备号不写入订单或服务关系。日报按“护工 + 服务时间段”命中设备绑定历史，再查询证据。
5. 绑定缺失、同一日报窗口出现多台设备或没有可用证据时，不生成正式日报，也不向家属推送。

### 3.4 平安健康险样例映射

## 4. 被护理人主档案

### 4.1 表名

`care_recipient`

### 4.2 职责

`care_recipient` 保存被护理人的稳定主档案，线上订单、线下订单、服务关系和护理日报都关联它。

字段清单：

| 字段             | 类型      | 说明                                 |
| ---------------- | --------- | ------------------------------------ |
| `id`             | `int`     | 主键                                 |
| `owner_user_id`  | `int`     | 所属 C 端用户 ID；纯线下可为空或后补 |
| `name`           | `varchar` | 姓名                                 |
| `gender`         | `char(1)` | 性别 `0`男 `1`女 `2`未知             |
| `birthday`       | `date`    | 生日，可空                           |
| `id_card_no`     | `varchar` | 身份证号，可空                       |
| `phone`          | `varchar` | 本人电话，可空                       |
| `guardian_name`  | `varchar` | 主家属姓名                           |
| `guardian_phone` | `varchar` | 主家属手机号                         |
| `address`        | `varchar` | 默认地址，可空                       |
| `illness`        | `varchar` | 病情，可空                           |
| `height_cm`      | `int`     | 身高（厘米），可空                   |
| `weight_kg`      | `int`     | 体重（公斤），可空                   |
| `description`    | `text`    | 档案备注                             |

### 4.3 去重规则

线下来源常见问题是：被护理人本人没有手机号，来源单手机号通常是子女或联系人手机号。

去重建议：

1. 有身份证号时，优先按身份证号去重。
2. 无身份证号时，按"联系人手机号 + 被护理人姓名 + 性别"做候选匹配。
3. 不要只按联系人手机号去重，因为一个联系人可能为多位家属下单。
4. 不要只按姓名去重，因为同名风险高。
5. 无法唯一确认时，标记为"待确认/疑似重复"，允许运营在后台合并档案。

线下订单也应保存被护理人快照，建议统一为 `recipient_snapshot(jsonb)`；如当前阶段先不单独建 JSON 字段，至少保留：

1. `recipient_name_snapshot`
2. `recipient_gender_snapshot`
3. `recipient_age_snapshot`

原因：被护理人档案后续可能被修改，历史订单必须还原当时来源单内容。

### 4.4 当前归属人与权限规则

`care_recipient.owner_user_id` 只表示“当前归属的 C 端用户”，是 C 端查看权限的唯一主依据。

规则：

1. C 端所有“我的被护理人 / 我的日报 / 我的订单”查询，统一以 `care_recipient.owner_user_id = 当前登录用户ID` 作为权限主条件。
2. `care_relation.owner_user_id`、`care_daily_report.owner_user_id` 如果保留，只作为生成时的冗余快照字段，不作为权限判断主依据。
3. 线下反向激活时允许直接覆盖 `care_recipient.owner_user_id`。
4. 覆盖完成后，旧 owner 立即失去该被护理人的全部历史查看权限，不保留“历史共享可见”口径。
5. 每次 owner 覆盖都必须强制记录变更日志，至少包含：`recipient_id`、`old_owner_user_id`、`new_owner_user_id`、`changed_at`、`changed_by`、`change_source`、`remark`。

## 5. 线下服务地点设计

住院陪护不能只保存一个地址文本。医院、科室、病区、楼层、床位是派单、到岗、统计的关键字段。

### 5.1 字段策略

当前线下订单 `care_offline_order` 服务地点相关字段：

| 字段                  | 说明          |
| --------------------- | ------------- |
| `institution_name`    | 医院/机构名称 |
| `institution_address` | 医院/机构地址 |
| `department_name`     | 科室          |
| `bed_no`              | 床位号        |

后续可扩展的字段：

- `service_place_type` — 服务地点类型 `hospital/home/nursing_home/other`
- `institution_id` — 机构主数据 ID
- `ward_area` — 病区
- `building_floor` — 楼栋/楼层
- `service_address_snapshot` — 完整服务地址快照

来源只给一段混合文本时，先保存完整快照（`institution_name` + `institution_address`），后续由运营补齐结构字段。

### 5.2 完整地址快照

完整地址建议由结构字段拼接：


## 7. 服务关系与改派

### 7.1 表名

`care_relation`

### 7.2 职责

服务关系是履约主轴，负责：

1. 护工与被护理人的实际服务绑定。
2. 预约时间到实际服务时间的承接。
3. 改派、换护工、多段服务的历史记录。
4. 护理日报归属。

### 7.3 字段清单

| 字段                  | 类型          | 说明                                         |
| --------------------- | ------------- | -------------------------------------------- |
| `id`                  | `int`         | 主键                                         |
| `relation_no`         | `varchar`     | 服务关系编号                                 |
| `source_type`         | `varchar`     | 来源类型 `online_order/offline_order/manual` |
| `source_order_id`     | `int`         | 来源订单 ID，`manual` 时可为空               |
| `owner_user_id`       | `int`         | 所属 C 端用户 ID，可后补                     |
| `care_recipient_id`   | `int`         | 被护理人 ID                                  |
| `caregiver_user_id`   | `int`         | 护工用户 ID，必填                             |
| `service_item_id`     | `int`         | 服务商品 ID，可空                            |
| `planned_start_at`    | `timestamptz` | 计划开始时间，可从订单带入                   |
| `planned_end_at`      | `timestamptz` | 计划结束时间，可从订单带入                   |
| `actual_start_at`     | `timestamptz` | 实际上单时间                                 |
| `actual_end_at`       | `timestamptz` | 实际下单时间                                 |
| `actual_service_days` | `int`         | 实际服务天数，结算确认时固化                 |
| `actual_service_hours`| `int`         | 实际服务小时，结算确认时固化                 |
| `service_status`      | `varchar`     | 服务状态                                     |

服务状态：

1. `pending` — 待服务
2. `serving` — 服务中
3. `completed` — 已完成
4. `cancelled` — 已取消

暂未实现的字段（后续按需扩展）：

- `sequence` — 同一订单下的关系序号
- `remark` — 备注
- `suspended` — 挂起状态（ICU、暂停护理等场景）

### 7.4 改派规则

一个订单允许产生多条 `care_relation`。常见原因包括护工中途换人、客户投诉、长单轮换、护工请假。

强约束：

1. 同一订单在同一时刻只能存在 1 条有效履约段，不允许时间重叠。
2. 同一被护理人在同一时刻只能存在 1 条 `serving` 状态的服务关系。
3. 当前阶段不支持“双护工同段协同服务”。
4. 新建 `serving` 关系前，必须先关闭已有的 `pending/serving` 有效关系。

改派流程：

1. 在同一个事务中锁定当前有效关系和订单。
2. 结束旧服务关系，写入 `actual_end_at`。
3. 按服务类型固化旧关系的 `actual_service_days` 或 `actual_service_hours`。
4. 旧关系状态改为 `completed` 或 `cancelled`。
5. 新建一条服务关系，写入新护工；新护工必须已有当前有效设备绑定。
6. 回写订单的 `caregiver_user_id` 为当前最新值。
7. 订单状态不因为旧关系结束而直接完成。
8. 同一天发生改派时，当天不创建正式护理日报、不推送家属通知；履约关系时间线保留改派事实，供运营处理。

注：`sequence` 字段（用于前端展示履约接力时间线）暂未实现，当前通过 `actual_start_at / actual_end_at` 排序。

### 7.5 实际服务天数与小时

预计服务天数/小时和实际服务天数/小时必须分开：

1. `care_offline_order.expected_service_days`：来源单预计天数，仅对 `day/month` 类服务生效。
2. `care_offline_order.expected_service_hours`：来源单预计小时，仅对 `hour` 类服务生效。
3. `care_relation.actual_service_days`：`day/month` 类履约段的实际服务天数，按自然日结算确认时固化。
4. `care_relation.actual_service_hours`：`hour` 类履约段的实际服务小时，结算确认时固化。
5. `care_online_order.actual_service_days`、`care_offline_order.actual_service_days`：由多个服务关系汇总回写，作为订单级展示和结算汇总口径。
6. `care_online_order.actual_service_hours`、`care_offline_order.actual_service_hours`：由多个服务关系汇总回写，作为小时类订单级展示和结算汇总口径。
7. `duration_type = 'hour'` 时，`actual_service_days = 0`，统一按小时收口，不做小数天折算。
8. 包月、长期单建议按自然月或服务周期拆分为多个履约段，不建议一条 `care_relation` 拉全年不分段。

日报与结算口径：

1. 用户侧正式日报默认在次日统一生成前一日内容，具体调度策略见本文第 9 章。
2. `day/month` 类服务按自然日累计。
3. 服务未走完整个计费日时，不自动记满 1 天，最终允许运营人工修正。

### 9.4 当前业务事实

结合当前业务，先承认以下事实：

1. 主流服务是按天服务，尤其是 24 小时护理服务。
2. 线下业务已经有“上单时间 / 下单时间”的概念。
3. 护工可以中途更换。
4. 智能工牌/硬件日报本质上是“按护理员按天产出”的。
5. 同一天换护工场景存在，但不是主流高频场景。
6. 正式护理日报是给用户/家属看的结果页，不是完整履约审计底账。

因此：

1. 履约事实和正式日报不必完全同一粒度。
2. 结算和追责必须依赖 `care_relation` 时间线。
3. 正式日报优先简单、稳定、可解释。

### 9.5 日报 V1 总体设计

V1 按三层模型理解：

1. 订单：交易与服务主线。
2. 履约关系：订单下的一段段实际服务记录。
3. 正式日报：订单的日级结果页。

核心结论：

1. 所有服务必须先有订单。
2. 所有 `care_relation` 都必须归属于某个订单。
3. 一个订单只对应一个被护理人。
4. 一个订单可以拆成多条履约关系，用于改派、交接、分段服务。
5. 正式护理日报按“订单 + 自然日”唯一生成。
6. 履约事实按 relation 保真，正式日报按订单对用户展示。

### 9.6 时间字段口径

`dispatched_at` 表示运营把订单分配给护工的时间，它是调度动作时间，不代表护工已经到岗，也不等于实际上单时间。

`actual_start_at` 表示护工实际开始服务的时间，它是真正开始履约的时间，也是结算和履约事实的重要依据。当前阶段如没有更精确的采集能力，可由运营人工确认；如必须临时兜底，可退化为使用派单时间，但只作为近似值，不作为长期规则。

`actual_end_at` 表示护工实际结束服务的时间，它可以是正常服务结束时间，也可以是中途交接时间或提前终止时间。

中途换护工时：

1. 旧 relation 的 `actual_end_at` = 交接时间。
2. 新 relation 的 `actual_start_at` = 交接时间。
3. 时间区间按左闭右开理解，即旧段 `[start, end)`、新段 `[end, next_end)`，避免逻辑重叠。

### 9.7 换护工处理规则 V1

派单后未开始服务就换人，这是“重新派单”，不是中途改派：

1. 旧 relation 标记 `cancelled`。
2. 不写 `actual_end_at`。
3. 新建新的 `pending` relation。
4. 订单状态保持 `pending_service`。

服务进行中途换人，这是正式改派：

1. 旧 relation 写 `actual_end_at`。
2. 旧 relation 固化实际服务天数/小时。
3. 旧 relation 标记 `completed`。
4. 新建新的 relation。
5. 如新护工立即接班，则新 relation 可直接进入 `serving`。
6. 如只是先改派未到岗，则新 relation 先保持 `pending`。
7. 订单状态通常保持 `in_service`。

V1 当前明确不支持：

1. 同一被护理人同一时刻双护工并行服务。
2. 一单多人。
3. 多 relation 并行共享同一时段。

### 9.8 日报生成规则 V1

日报粒度：

1. 护理日报按自然日生成，不按班次生成。
2. 同一订单同一天只生成一份正式护理日报。
3. 正式护理日报对外按“订单 + 日期”展示。

日报生成时间：

1. 每日固定时间运行定时任务，当前默认按早上 `07:00` 生成前一自然日的正式日报。
2. 例如 `2026-07-02 07:00` 生成 `2026-07-01` 的日报。

日报主来源护工判定：

1. 生成 `report_date` 当天的正式日报时，按 `report_date 23:59:59` 这一刻仍然有效的 relation 作为主 relation。
2. 主 relation 对应的护工作为主护工。
3. 正式日报优先使用这位主护工在 `report_date` 当天的硬件日报作为来源。

说明：

1. 不是“定时任务跑的时候系统当前护工是谁就用谁”。
2. 而是“历史上该自然日结束时，这张订单最后归属到谁”。

回退规则：

1. 如主护工当天没有硬件日报，则回退到该订单当天第一位有效护工的硬件日报。
2. 再找不到，则生成空壳正式日报并标记待人工补充。
3. 已提交微信退款后，退款状态优先走微信回调更新；如回调延迟或丢失，再由退款状态对账任务补偿查询。这属于支付退款状态收口，不改变日报生成主链路。

夜班与跨天规则：

1. V1 不单独做夜班日报。
2. 夜班数据仍按自然日拆分。
3. `20:00 - 次日 08:00` 的服务不会丢失日报，而是分别体现在两个自然日的正式日报里。
4. `7 月 1 日 20:00 - 23:59` 归入 `7 月 1 日` 日报。
5. `7 月 2 日 00:00 - 08:00` 归入 `7 月 2 日` 日报。

信息缺失处理态度：

1. 同一天中途换护工时，不生成正式护理日报，不发送通知。
2. 改派事实只在运营侧履约时间线中保留；如有必要，运营可后续人工补充家属说明。
3. 这是有意的保守策略：硬件原始数据按护理员按天产出，现阶段不做多护工证据融合，也不向家属展示不完整事实。

一句话总结：

1. 账按 relation 算。
2. 日报按订单出。
3. 原始来源按主护工选。

### 9.9 正式日报与结算的关系

正式日报不是结算底账。

V1 约定：

1. 结算以 relation 的 `actual_start_at` / `actual_end_at` 为准。
2. `day/month` 类服务按自然日累计。
3. `hour` 类服务按实际小时累计。
4. 正式日报只做用户侧结果展示，不作为唯一结算依据。

### 9.10 数据模型调整目标 V1

`care_relation` 建议继续收口：

1. `source_order_id` 改为非空。
2. `source_type` 去掉 `manual`。
3. 增加 `actual_service_hours`。
4. 后续建议增加 `sequence_no`，用于展示履约接力顺序。

`care_daily_report` 建议按订单维度改造为：

1. `source_type`
2. `source_order_id`
3. `primary_relation_id`
4. `relation_ids_snapshot`
5. `care_recipient_id`
6. `caregiver_user_id`
7. `report_date`
8. `source_badge_report_id`
9. `summary_text`
10. `report_card`

唯一键建议改为：

1. `(source_type, source_order_id, report_date)`

订单表继续保留：

1. `caregiver_user_id` 作为当前/最新护工缓存。
2. `actual_service_days`
3. `actual_service_hours`

完整履约历史不放在订单主表，仍以 relation 为准。

### 9.11 迁移实施切片

第 1 切片：数据模型与迁移

1. `care_relation.source_order_id` 改为非空，移除 `manual` 来源及其 DTO/API 分支。
2. 增加 `actual_service_hours`，补齐线上、线下订单的实际时长汇总字段。
3. 将 `care_daily_report` 改为订单级字段和唯一键 `(source_type, source_order_id, report_date)`；迁移时保留 `primary_relation_id` 和 `relation_ids_snapshot`。
4. 为同订单有效履约段、同被护理人 `serving` 段增加可并发保障的数据库约束或等价锁定方案。

第 2 切片：履约与订单状态

1. 建立单一“改派”应用服务：锁定订单和有效关系、关闭旧段、固化时长、创建新段、更新订单当前指派缓存，整个过程必须在一个事务内完成。
2. 线上、线下订单都只通过该服务创建或改派履约关系，禁止各自复制关闭/创建逻辑。
3. 订单状态由履约动作统一驱动；最后一段有效关系结束后才允许订单完成。

第 3 切片：订单级日报

1. 用统一任务扫描线上和线下订单，按订单和自然日去重生成正式日报。
2. 按 `report_date 23:59:59` 判定主 relation 和主护工；主来源缺失时按本章回退规则生成待审核日报，不能直接生成可推送的正式内容。
3. 用户端列表、详情、周趋势及管理后台查询改用订单级日报字段；权限查询联接 `care_recipient`，按当前 `owner_user_id` 判断。

第 4 切片：历史数据、页面与测试

1. 回填旧日报和清洗旧 relation 后，再启用新唯一键；无法自动归并的数据进入人工审核清单。
2. 后台订单详情展示 relation 时间线，并提供显式改派操作；日报页面只展示订单级正式日报。
3. 至少覆盖：护工有效设备绑定校验、并发改派、同日改派不出日报、夜班跨天、线上/线下统一任务、证据缺失不推送、当前归属权限变更、日报去重和任务重跑幂等。

### 9.12 实施边界

本文档当前只定 V1，不在本阶段展开：

1. 多护工日报融合。
2. 班次级日报。
3. 一单多人。
4. 双护工并行。
5. 复杂 AI 角色识别。
6. 报表级复杂结算编排。

## 12. 线下录单页面建议

录单顺序：

1. 基础信息：外部工单号、订单状态、结算状态。
2. 被护理人：搜索已有档案，或点击"新建"快速录入姓名/性别/年龄/手机号。选中后展示性别、年龄（只读）。
3. 家属信息：家属姓名、家属电话。
4. 服务信息：服务商品、预约到岗时间、预计天数、自理状况、身高体重、病情描述。
5. 服务地点：医院/机构、地址、科室、床位号。
6. 派单信息：**委派护工**、护工当前设备绑定状态、派单时间。

说明：
- 委派护工和设备号在同一行并排，点击弹出独立选择对话框（护工用表格搜索，设备号展示 `status=IDLE` 的空闲设备列表）
- 两者互斥：选护工自动清空设备号，选设备号自动清空护工
7. 备注。

说明：

- 线下表单不做区块分割，所有字段平铺展示。
- 被护理人支持搜索已有档案和快速新建两种方式。
- 选中已有被护理人后，自动填入家属姓名和家属电话。

列表默认展示：

1. 外部订单号
2. 被护理人
3. 服务地点
4. 预约到岗时间
5. 预计/实际天数
6. 护工
7. 订单状态
8. 结算状态

服务地点结构字段应允许后补和修正。运营可以先用完整地址快速录入，服务进行中或结算前再补齐医院、科室、床位号。

## 13. 最终结论

1. 线上和线下必须分表，线上是服务商品订单，线下是机构/保险来源服务单。
2. 线上订单必须同时保留 `service_item_snapshot(jsonb)` 与 `recipient_snapshot(jsonb)` 两类快照，但金额、数量、商品 ID、状态、预约时间必须保留结构化列。
3. 线下订单保存医院/机构、科室、床位等结构字段，同时允许先保存完整地址再后补结构字段。
4. 被护理人档案只保存人的稳定信息；`care_recipient.owner_user_id` 只表示当前归属人，C 端权限统一按它判断，覆盖 owner 时必须记变更日志。
5. 线下被护理人去重优先身份证号；没有身份证号时只做候选匹配，无法唯一确认则允许运营合并档案。
6. 一个订单可以有多条服务关系，但同一订单、同一被护理人在同一时刻只能有 1 条有效履约段；改派必须在一个事务里完成“关旧开新”。
7. 正式护理日报按“同一订单 + 同一自然日”聚合生成；具体生成规则、主护工判定与跨天处理见本文第 9 章。
8. `day/month` 类服务按自然日累计 `actual_service_days`；`hour` 类服务统一记录 `actual_service_hours`，不折算小数天。
9. 线下订单状态机收口为 5 个状态：`pending_dispatch`、`pending_service`、`in_service`、`completed`、`cancelled`。
10. `banners` / `detail` 等多图字段在数据库中存为逗号分隔字符串，前端按需转为数组使用；服务商品基准价统一使用整型分字段 `base_price_amount`。
11. 以下字段/功能为待扩展实现：ward*area、building_floor、service_place_type、institution_id、service_address_snapshot、source_channel_name、source_created_at、source_payload、collaborator_name_snapshot、collaborator_user_id、核销字段（verification*_）、sequence、remark、service_relation 的 suspended 挂起状态。
