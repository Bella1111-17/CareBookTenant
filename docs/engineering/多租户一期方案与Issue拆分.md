# 多租户一期方案与 Issue 拆分

## 1. 背景

CareBook 当前是单租户系统，但已经确认要提高多租户优先级。第一阶段不是把整套护理业务全面 SaaS 化，而是先围绕智能工牌相关的 ToB 分支做一个可试点、可控、可维护的最简多租户能力。

这条 ToB 分支当前只聚焦机构侧的设备类业务，不追求一次性把完整后台能力都开放给租户。

## 2. 一期共识

### 2.1 范围

第一期优先做“租户表 + 租户用户隔离 + 智能工牌业务隔离 + 租户管理可用”这条主线，不做全系统租户化。

### 2.2 当前真正隔离的数据

一期先把租户隔离重点放在以下两个主隔离点：

- 用户表 `sys_user`
- 智能工牌相关业务表

其中智能工牌相关业务表包括：

- 设备表
- 设备绑定关系表
- 录音记录表
- GPS / 事件日志表
- 后续工牌分析结果表

一期暂不要求把所有系统表都做成严格租户化，但要求：

- 租户只能查询和管理自己租户下的用户
- 租户只能查询自己租户下的设备、录音、绑定、日志数据
- 平台可以跨租户查看并管理

### 2.3 菜单与权限口径

一期不走“租户类型自动决定菜单”的复杂方案，统一采用：

- 租户只负责数据归属隔离
- 菜单完全由角色决定
- 角色由平台统一规划
- 租户不自行设计角色体系

也就是说：

- 给用户什么角色，用户就看什么菜单
- 给租户管理员角色绑定什么菜单，租户管理员就只能看到这些菜单

### 2.4 当前租户角色策略

一期先只保留一个租户预设角色：

- `tenant_admin`：机构管理员

后续如果业务需要，再扩展：

- `tenant_operator`
- `tenant_auditor`

但这些不是一期必做项。

### 2.5 平台注册用户口径

平台注册或新创建的普通账号，默认：

- 不自动授予业务角色
- 不自动授予菜单权限
- 可以理解为“有账号但没有业务权限”

这样更安全，后续由平台手工授予角色。

### 2.6 租户用户登录口径

租户用户不进入平台首页。

一期规则：

- 平台用户登录后进入平台首页
- 租户用户登录后直接进入业务菜单
- 当前默认跳到设备管理页

### 2.7 当前租户能力边界

一期先尽量简单。

当前租户用户一期至少需要：

- 登录后台
- 使用租户管理能力维护本机构账号
- 查看自己机构的设备类业务数据

当前暂不做复杂开放：

- 自定义角色设计
- 自定义菜单配置
- 机构侧首页
- 大范围租户子后台定制

## 3. 技术方案

### 3.1 总体方案

采用：

- 共享数据库
- 共享 Schema
- `tenant_id` 字段隔离
- 应用层软隔离
- 角色驱动菜单

不采用：

- Prisma 重构
- PostgreSQL Schema 隔离
- PostgreSQL RLS 作为一期核心隔离手段
- 第三方 TypeORM 多租户插件

### 3.2 一期的核心思想

一期多租户不是“整套系统完整 SaaS 化”，而是：

- 先让不同机构的数据不串
- 再通过角色控制这些机构能看到哪些菜单

所以必须分清两层：

- `tenant_id`：解决“数据归谁”
- `role + menu`：解决“这个人能看什么”

### 3.3 为什么先做这种最简模型

当前阶段目标是试点可用，不是做一个过重的租户平台。

如果现在就做：

- 强数据库隔离
- 租户自定义角色体系
- 租户自定义菜单体系
- 多层级组织权限

会显著增加实现成本和排障复杂度，不适合当前推进节奏。

## 4. 数据与身份建模

### 4.1 用户身份域

系统区分两类账号：

- 平台账号：`user_scope = platform`
- 租户账号：`user_scope = tenant`

平台账号用于平台运营和全局管理。  
租户账号用于机构内部访问本机构业务。

### 4.2 用户表策略

继续共用一张 `sys_user` 表，并保留：

- `tenant_id`
- `user_scope`

规则如下：

- 平台账号 `tenant_id` 可为空
- 租户账号 `tenant_id` 必填
- 有效账号 `user_name` 全局唯一
- 租户查询用户列表时，只能查到自己 `tenant_id` 下的用户
- 租户编辑、停用、分配角色时，也只能作用于自己 `tenant_id` 下的用户

### 4.3 角色表策略

继续共用一张 `sys_role` 表，并保留：

- `tenant_id`

规则如下：

- 平台角色 `tenant_id` 可为空
- 租户角色 `tenant_id` 绑定具体租户
- 一期租户角色由平台统一预设

### 4.4 业务表策略

工牌相关业务表带 `tenant_id`，作为当前主要业务隔离边界。

建议一期直接统一到租户业务基类上，至少覆盖：

- `badge_device`
- `device_user_binding`
- `audio_record`
- `device_event_log`
- `device_gps_log`

## 5. 核心实现原则

### 5.1 数据隔离只靠应用层

一期所有租户隔离以应用层为准，不依赖数据库强隔离。

要求包括：

- 请求上下文里有 `tenantId` / `userScope` / `userId`
- 查询 helper 自动补租户条件
- 平台用户允许跨租户筛选
- 租户用户强制只查本租户
- 用户管理模块同样纳入租户过滤，而不是只做设备数据过滤

### 5.2 菜单完全由角色决定

平台不按“租户类型”动态拼菜单。

统一规则：

- 角色绑定菜单
- 用户绑定角色
- 后端根据角色返回菜单树

租户用户看不到什么，不是因为“他是租户”，而是因为“他的角色没有那个菜单”。

### 5.3 租户当前默认角色

平台创建租户管理员时，默认只初始化：

- 一个租户管理员角色 `tenant_admin`
- 一个租户管理员账号

当前只给它绑定最小业务菜单，并允许它管理本租户用户。

### 5.4 平台注册默认无权限

平台注册后的普通用户默认不自动拥有角色和菜单。

只有平台显式授予角色后，用户才具备业务访问能力。

### 5.5 租户无首页

租户用户不使用平台首页，不进入 `/index`。

默认跳转到：

- `/badge/binding`

即设备管理页。

## 6. 一期菜单策略

### 6.1 平台用户

平台用户保留原有平台首页和平台菜单：

- 首页
- 租户管理
- 平台工牌总览
- 平台录音总览
- 其他平台控制面菜单

### 6.2 租户用户

租户用户当前只看与工牌业务相关的角色菜单。

建议的一期默认最小菜单：

- 租户管理
- 智能工牌目录
- 设备管理

如果平台后续需要，也可以继续给 `tenant_admin` 角色补：

- 录音记录
- GPS 日志
- 设备事件

但是否可见，以平台给这个角色绑定的菜单为准。

### 6.3 当前不开放给租户的菜单

一期默认不开放给租户：

- 平台首页
- 租户管理
- 平台级用户管理
- 平台级角色管理
- 护理日报主链路

## 7. 当前实现路线

### Issue 1: 固化最简多租户边界

目标：
把“数据隔离优先、菜单由角色决定、平台注册默认无权限”的口径固化下来。

### Issue 2: 修正租户身份与登录默认落点

目标：
让租户用户不进入平台首页，登录后直接进入业务菜单。

### Issue 3: 建立租户管理与租户管理员模板

目标：
平台可以进入租户管理页面；初始化租户时，默认创建 `tenant_admin` 角色与租户管理员账号，并绑定最小业务菜单。

### Issue 4: 落地用户与工牌业务数据隔离

目标：
`sys_user`、设备、录音、GPS、事件等数据按 `tenant_id` 隔离，平台可筛全量，租户只看自己。

### Issue 5: 平台注册默认无权限

目标：
平台注册账号后默认无业务角色、无业务菜单，必须由平台显式授权。

### Issue 6: 历史试点数据归属回填

目标：
把试点机构的设备和录音挂到对应租户下，保证租户登录后能看到真实业务数据。

## 8. 一期完成定义

满足以下条件即可视为一期最简多租户能力完成：

- 平台可以创建租户和租户管理员
- 平台可以使用租户管理页面
- 平台可以通过角色给租户管理员分配菜单
- 平台普通注册用户默认无业务权限
- 租户用户登录后不进入平台首页
- 租户用户只能查询和管理自己租户下的用户
- 租户用户只能看到自己角色对应的业务菜单
- 设备、录音等主要工牌业务数据完成租户隔离
- 平台可跨租户查看，租户只能看自己
